# 产品需求文档：AFramework v1.1 跨项目复用收口设计

> 版本: v1.1 | 创建时间: 【2026-04-13 14:58:59】 | 状态: confirmed | 时间格式: 本地时间 `【YYYY-MM-DD HH:MM:SS】`

## 1. 项目概述

### 1.1 背景

当前 `AFramework` 已经具备 `Core`、`Event`、`FSM/Procedure`、`Assets`、`UI`、`Hotfix`、`Audio`、`StepSystem` 等能力，能够在现有项目中稳定运行。但从“跨项目复用”的角度看，仍存在以下问题：

- 框架能力与宿主项目能力边界不够清楚，`Assets/LWFramework` 与 `Assets/Scripts` 的职责容易混淆。
- 启动链路以宿主 `Startup` 直接注册模块为主，适合当前项目，但对外部项目接入说明不够标准化。
- `Hotfix` 已经收敛到 `ByCode`、`Reflection` 为主，但还缺少明确的对外口径与接入约束。
- `StepSystem` 价值很高，但并不适合作为所有项目的默认核心模块，更适合转为业务插件能力。
- 仓库已经有较多工程化积累，但还没有形成一份面向“跨项目接入”的统一设计说明。

本版本不追求彻底重构或一次性完成标准 UPM 包化，而是采用“保守模块化”路线：在不破坏现有项目结构和运行方式的前提下，先完成框架边界、模块职责、接入规则、插件定位和文档口径的统一，推动 `AFramework` 从“当前项目可用”升级为“跨项目可复用”。

### 1.2 目标

- 采用“保守模块化”路线完成 `AFramework v1.1` 设计收口，优先稳定现有结构而非大规模迁移目录。
- 明确 `Core + Event + FSM/Procedure + Assets + UI + Hotfix` 为默认内置核心能力。
- 将 `StepSystem` 从默认核心能力中剥离，定义为业务插件。
- 保留 `Hotfix` 核心能力定位，并明确默认使用 `ByCode`，可选使用 `Reflection`。
- 约束 `Reflection` 模式统一通过 `Assets` 资源系统，从固定路径加载对应 DLL 文件。
- 保持宿主项目负责 `Startup`、业务 `Procedure`、业务 `UI`、业务资源和业务插件启用逻辑。
- 让当前仓库具备 `Git / UPM` 方向的接入准备能力，即 `UPM-ready`，但不要求本版本彻底完成标准包化。

### 1.3 非目标

- 不在 `v1.1` 中执行彻底的 UPM 包拆分或多仓库拆分。
- 不在 `v1.1` 中大规模迁移当前目录结构。
- 不新增第三条正式热更路线，不将 `HybridCLR` 作为本版本对外正式能力。
- 不把 `StepSystem` 重新拉回核心底盘。
- 不把本版本变成全面重构版，不解决所有历史设计问题。

## 3. 功能需求

### 3.1 保守模块化收口

**描述**: `v1.1` 采用保守模块化路线。逻辑上完成模块边界梳理，但物理目录尽量保持现状，避免对当前项目造成大范围破坏。

**用户故事**:
- 作为框架维护者，我希望先把框架职责和接入规范说清楚，而不是直接大拆目录，以便低风险推进跨项目复用。

**验收标准**:
- [ ] 对外明确 `v1.1` 为“保守模块化”版本，而非彻底包化版本。
- [ ] 框架核心、宿主项目、业务插件三层边界有统一定义。
- [ ] 现有目录结构原则上保持不变，仅新增必要文档和后续收口约束。

**优先级**: P0

### 3.2 核心能力边界确定

**描述**: `Core + Event + FSM/Procedure + Assets + UI + Hotfix` 被定义为默认内置核心能力，其中 `FSM/Procedure` 指的是流程机制和基础能力，不包含具体业务流程状态。

**用户故事**:
- 作为宿主项目开发者，我希望一眼看清什么属于框架必须内置的能力，以便判断最小接入范围。

**验收标准**:
- [ ] 文档中明确核心能力列表。
- [ ] 核心能力定义中不混入宿主项目的业务状态、业务界面和业务资源。
- [ ] `Hotfix` 被纳入核心能力，而 `StepSystem` 不纳入核心能力。

**优先级**: P0

### 3.3 宿主项目职责收口

**描述**: 宿主项目负责 `Startup/Bootstrap`、具体业务 `Procedure`、业务 `UI/View`、业务资源与场景、项目配置和业务插件启用逻辑。

**用户故事**:
- 作为宿主项目开发者，我希望接入框架时不需要猜测哪些代码应该继续放在项目里。

**验收标准**:
- [ ] 文档明确 `Assets/Scripts` 代表宿主项目示例/业务层，而不是框架核心层。
- [ ] 宿主项目负责决定首个 `Procedure`、模块启用方式和项目资源配置。
- [ ] 宿主项目负责决定是否启用 `StepSystem` 以及如何提供插件资源。

**优先级**: P0

### 3.4 启动链路与模块启停规范

**描述**: 保留宿主 `Startup` 作为总入口，但从“全部硬编码注册”升级到“宿主入口 + 模块配置驱动”的约束模型。

**用户故事**:
- 作为框架维护者，我希望不同项目能够在不改框架底层的前提下调整启用模块和启动顺序。

**验收标准**:
- [ ] 宿主项目保留 `Startup` 作为装配入口。
- [ ] 文档中定义模块启停、初始化顺序、默认行为和失败策略。
- [ ] `StepSystem` 默认不进入核心启动链，只有宿主显式启用时才注册。
- [ ] `Core / Assets / FSM` 初始化失败默认阻断启动。
- [ ] `UI / Hotfix / 插件模块` 初始化失败可按配置定义为阻断或降级继续。

**优先级**: P0

### 3.5 Hotfix 路线收敛

**描述**: `Hotfix` 作为框架核心能力继续保留，但正式对外路线只保留 `ByCode` 与 `Reflection`。默认使用 `ByCode`；如项目选择 `Reflection`，则统一通过 `Assets` 从固定路径加载 DLL 文件。

**用户故事**:
- 作为宿主项目开发者，我希望热更规则足够简单，不会误解为“所有项目都必须做代码更新”。

**验收标准**:
- [ ] 默认热更模式为 `ByCode`。
- [ ] `Reflection` 为可选路线，而非默认路线。
- [ ] `Reflection` 使用 `Assets` 资源系统从固定路径加载对应 DLL 文件。
- [ ] 文档清楚解释“热更能力存在”与“项目实际是否更新代码”是两回事。
- [ ] `HybridCLR` 不作为 `v1.1` 正式对外能力。

**优先级**: P0

### 3.6 StepSystem 插件化

**描述**: `StepSystem` 在 `v1.1` 中被定义为业务插件，不再要求所有项目默认携带。其运行时代码、编辑器能力、文档和测试可以继续增强，但定位上属于业务插件层。

**用户故事**:
- 作为业务插件开发者，我希望 `StepSystem` 能单独接入、单独维护，而不是绑定所有项目默认存在。

**验收标准**:
- [ ] 文档中明确 `StepSystem` 为业务插件，而非核心底盘。
- [ ] 核心框架不反向依赖 `StepSystem`。
- [ ] 未启用 `StepSystem` 时，核心链路仍能稳定运行。
- [ ] `StepSystem` 可依赖 `Core / Event / Assets / UI` 等基础能力。

**优先级**: P0

### 3.7 跨项目接入与 UPM-ready 约束

**描述**: 本版本不彻底完成 UPM 包化，但框架要具备向 `Git / UPM` 方向迁移的准备条件，包括边界清晰、接入说明明确、文档统一。

**用户故事**:
- 作为框架维护者，我希望新项目后续能通过 Git/UPM 方向逐步接入，而不是每次手工复制时都重新整理一遍边界。

**验收标准**:
- [ ] 新增统一接入文档，说明框架核心、宿主项目、业务插件三层职责。
- [ ] 接入方式明确以 `Git / UPM` 为后续目标，但本版本不承诺彻底包化。
- [ ] 文档提供最小可运行宿主项目的接入说明。
- [ ] 现有文档中与实现冲突的口径被标记为后续需要统一修订。

**优先级**: P0

## 4. 非功能需求

### 4.1 技术约束

- 运行环境：Unity `2022.3.62f3`
- 文档语言：中文
- 代码注释要求：新增代码必须包含函数级中文注释
- 代码风格：遵循仓库现有命名与目录约定，不使用 `var`
- 架构约束：沿用现有 `Startup + MainManager + ManagerUtility + 各模块 Manager` 总体模式，在此基础上收口

### 4.2 性能要求

- `v1.1` 设计不能引导出明显增加运行时初始化成本的方案。
- 默认 `ByCode` 路线必须保持最小额外负担。
- `Reflection` 只在宿主显式启用时进入固定路径装载流程。
- 插件未启用时不应给核心链路增加额外开销。

### 4.3 安全要求

- `Reflection` 仅通过受控固定路径和 `Assets` 链路装载 DLL。
- 热更模式切换必须是宿主项目显式配置行为。
- 框架核心不应因为业务插件缺失而崩溃。

## 5. 技术架构概览

`AFramework v1.1` 继续保留现有运行时/编辑器总体组织方式，但在逻辑上划分为三层：

- 框架核心层：`Core`、`Event`、`FSM/Procedure`、`Assets`、`UI`、`Hotfix`
- 宿主项目层：`Startup`、业务 `Procedure`、业务 `View`、项目资源、项目配置
- 业务插件层：`StepSystem` 以及未来类似的可选插件

### 5.1 系统架构图

- `Startup` 由宿主项目提供，负责总装配与模块启停。
- `MainManager` 和各基础 `Manager` 仍构成主运行时骨架。
- `Assets` 为资源与反射 DLL 加载提供统一入口。
- `Hotfix` 默认走 `ByCode`，可选切换到 `Reflection`。
- `StepSystem` 通过宿主项目决定是否接入，不进入默认核心链路。

### 5.2 核心技术选型

| 层次 | 技术 | 理由 |
|------|------|------|
| 运行时骨架 | 现有 `MainManager + ManagerUtility` 模式 | 保持兼容，降低 v1.1 风险 |
| 资源管理 | `LWAssets` | 已内建，且是 `Reflection` DLL 固定路径加载的基础 |
| 热更能力 | `ByCode + Reflection` | 满足当前项目需求，收敛维护面 |
| 流程能力 | `FSM/Procedure` | 继续作为核心流程机制 |
| 插件能力 | `StepSystem` | 定位为业务插件，按需接入 |

## 6. 关键设计决策

| 决策 | 选择 | 替代方案 | 选择理由 |
|------|------|---------|---------|
| 模块化路线 | 方案 A：保守模块化 | 标准模块化底盘 / 平台化插件框架 | 当前更重视低风险跨项目复用，而不是立即大拆 |
| 最小核心层 | `Core + Event + FSM/Procedure + Assets + UI` | 更薄核心 / 更厚核心 | 兼顾现有架构和最小可运行能力 |
| Hotfix 位置 | 核心能力 | 可选插件 / 增强模块 | 用户明确要求保留核心地位 |
| Hotfix 默认路线 | `ByCode` | 默认 `Reflection` | 更稳定，且并非所有项目都做代码更新 |
| Reflection 装载方式 | 通过 `Assets` 从固定路径加载 DLL | 独立下载/装载链路 | 与现有资源系统统一，维护成本更低 |
| StepSystem 位置 | 业务插件 | 核心底盘 / 官方增强模块 | 并非所有项目都需要，适合作为按需接入插件 |
| 跨项目接入方式 | 以 `Git / UPM` 为目标，先做到 `UPM-ready` | 直接彻底包化 | 当前版本不做大规模迁移，先收口再推进 |
| 框架包内容 | 只放通用底层 | 连同业务模板一起打包 | 避免宿主项目与框架边界继续混淆 |

### 8.2 三方分析摘要

- 本次未启用子代理，直接基于用户连续确认结果形成单方案收口。
- 关键共识：
  - 采用保守模块化
  - 核心层不继续扩厚
  - `Hotfix` 核心化但默认 `ByCode`
  - `StepSystem` 插件化
  - 后续方向面向 `Git / UPM`

### 8.3 综合方案与最终决策

| 方案 | 偏向 | 优势 | 劣势 | 决策结果 |
|------|------|------|------|---------|
| 方案 A：保守模块化 | 低风险演进 | 兼容当前项目、落地快、适合逐步跨项目复用 | 物理包边界不会在本版完全成型 | 采纳 |
| 方案 B：标准模块化底盘 | 中期结构清晰 | 更利于长期维护与包化 | 需要更大改动和目录重整 | 本版否决 |
| 方案 C：平台化插件框架 | 长期平台化 | 理论上最灵活 | 当前投入过大，超出本版目标 | 本版否决 |

### 8.4 关键 Tradeoff 备忘

1. **模块化路线**
   - 选择了: 保守模块化
   - 放弃了: 立即标准包化
   - 放弃理由: 当前更重视低风险跨项目复用
   - 未来可能回头的条件: 当 `v1.1` 接入规则稳定后，再推进更彻底的 UPM 包化

2. **Hotfix 默认行为**
   - 选择了: 默认 `ByCode`
   - 放弃了: 默认 `Reflection`
   - 放弃理由: 并非所有项目都需要实际代码更新
   - 未来可能回头的条件: 若未来多数项目都稳定依赖反射装载，再重新评估默认路线

3. **StepSystem 位置**
   - 选择了: 业务插件
   - 放弃了: 核心底盘
   - 放弃理由: 不具备所有项目默认必需性
   - 未来可能回头的条件: 若后续框架明确聚焦“教学/引导型项目专用底盘”，可重新评估其核心化价值

## 9. 验收与非目标

### 9.1 验收标准

- 核心链路的设计边界被统一说明，并与宿主层、插件层形成清晰职责划分。
- `Hotfix` 对外规则明确为“默认 `ByCode`，按需 `Reflection`，且 `Reflection` 统一从固定资源路径加载 DLL”。
- `StepSystem` 被明确定义为业务插件，不再作为默认核心模块。
- 文档能够指导新项目理解最小接入范围、模块归属和后续 `Git / UPM` 方向。

### 9.2 非目标重申

- 不承诺在本版完成彻底包化交付。
- 不承诺在本版完成目录重构。
- 不承诺在本版补齐所有历史问题。
