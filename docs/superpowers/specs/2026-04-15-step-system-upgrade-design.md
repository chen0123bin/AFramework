# 产品需求文档：StepSystem 图编辑器、Action 与示例一体化升级设计

> 版本: v2.0-design | 创建时间: 2026-04-15 09:54:57 | 状态: confirmed | 时间格式: `yyyy-MM-dd HH:mm:ss`

---

## 1. 项目概述

### 1.1 背景

当前仓库内的 `StepSystem` 已经具备以下基础能力：

- 运行时支持 XML 驱动的 DAG 步骤图、节点动作执行、前进/后退/跳转补齐。
- 编辑器支持节点编辑、连线编辑、XML 导入导出、基础校验、PlayMode 预览。
- 工程内已有 `StepActionTest.xml`、`StepStage4Test.xml` 等示例，能够覆盖部分条件分支、并行执行与动作能力。

但从“继续演进成可长期维护的通用流程编排系统”的角度看，当前实现仍存在明显短板：

- `Graph` 视图信息密度较低，节点和连线的关键语义需要依赖右侧面板才能理解。
- 编辑器主要集中在 `StepEditorWindow.cs` 中，视图、参数表单、运行时联调和数据逻辑耦合较重。
- `Action` 体系当前主要依赖 `Type.FullName + StepParam`，对编辑器来说“能识别”，但不够“可展示、可搜索、可分类”。
- XML 导入兼容属性参数与 `<param>` 参数两种写法，但导出规范尚未完全收口，长期会增加维护成本。
- 现有示例更偏“历史测试样本”，缺少面向使用者的分层示例体系，难以帮助快速上手和回归验证。

本次升级不把 `StepSystem` 改造成独立资产格式系统，也不引入新的主数据源。核心目标是在保留 XML 作为主数据源的前提下，重构编辑器内部结构，完成 `Graph` 体验升级、`Action` 扩展体系升级和示例体系升级，使 `StepSystem` 从“可用工具”提升为“可持续演进的流程编排工具”。

### 1.2 目标

- 保持 XML 作为 `StepSystem` 的唯一主数据源，编辑器围绕 XML 工作。
- 重构编辑器内部模型与渲染层，降低 `StepEditorWindow` 的职责耦合。
- 明显增强 `Graph` 的信息表达能力、编辑效率和运行时联调体验。
- 建立 `Action` 元数据体系，使新增 `Action` 可被编辑器自动分类、展示、摘要和参数化。
- 增加一批面向通用流程编排的常用 `Action`，并保留后续接入 `LWFramework` 能力的扩展空间。
- 建立分层示例体系，使示例既能作为文档，又能作为导入导出和运行时回归样本。

### 1.3 非目标

- 不引入新的主数据资产格式，不以 `ScriptableObject Graph Asset` 取代 XML。
- 不在本次设计中实现子图、嵌套图、组合节点、时间轴节点等大范围平台化能力。
- 不引入脚本表达式语言或复杂条件 DSL，连线条件仍维持轻量条件模型。
- 不要求一次性补齐所有业务专用 `Action`，优先补齐通用动作和少量框架桥接动作。
- 不要求本次迭代完成全部 UI 美术化，仅要求功能性与可读性明显提升。

## 2. 用户画像

| 用户角色 | 描述 | 核心诉求 |
|---------|------|---------|
| 框架维护者 | 维护 `StepSystem` 运行时、编辑器与文档的开发者 | 希望结构清晰、可扩展、后续新增能力成本可控 |

## 3. 当前系统分析

### 3.1 现有代码结构

当前 `StepSystem` 相关代码主要分布在以下位置：

- 运行时：
  - `Assets/LWFramework/RunTime/StepSystem/StepManager.cs`
  - `Assets/LWFramework/RunTime/StepSystem/IO/StepXmlLoader.cs`
  - `Assets/LWFramework/RunTime/StepSystem/Graph/`
  - `Assets/LWFramework/RunTime/StepSystem/Action/`
- 编辑器：
  - `Assets/LWFramework/Editor/StepSystem/StepEditorWindow.cs`
  - `Assets/LWFramework/Editor/StepSystem/GraphView/StepGraphView.cs`
  - `Assets/LWFramework/Editor/StepSystem/GraphView/StepNodeView.cs`
  - `Assets/LWFramework/Editor/StepSystem/StepXmlImporter.cs`
  - `Assets/LWFramework/Editor/StepSystem/StepXmlExporter.cs`
- 示例与接入：
  - `Assets/0Res/RawFiles/StepActionTest.xml`
  - `Assets/0Res/RawFiles/StepStage4Test.xml`
  - `Assets/Scripts/Procedure/StepProcedure.cs`

### 3.2 当前系统优势

- 运行时基础闭环完整，`StepManager`、`StepGraph`、`StepNode`、`BaseStepAction` 的职责边界已经初步建立。
- `StepParamAttribute` 已经提供参数绑定能力，为后续 `Action` 扩展留出了统一入口。
- 编辑器能够支持 XML 导入导出与基础联调，不需要从零开始搭工具。
- 当前示例已经覆盖条件分支、优先级、并行动作、音频、动画、对象控制等关键场景。

### 3.3 当前主要问题

#### 3.3.1 Graph 表现问题

- 节点标题当前几乎只显示 `Id`，节点名称、执行模式、动作摘要无法直接从图上读取。
- 连线不显示条件和优先级，分支逻辑必须依赖右侧 Inspector 逐条查看。
- 校验结果只在提示层面给出，图中缺少错误定位与视觉提醒。
- 当前运行态高亮只覆盖节点状态，无法表达当前 `Action`、执行路径和等待原因。

#### 3.3.2 编辑效率问题

- 节点新增、动作新增、动作排序、节点布局等操作偏基础，处理稍复杂图时效率不足。
- `Action` 选择器直接暴露完整类型名，对使用者不友好。
- 编辑器内部职责集中在单窗口中，后续继续叠加功能会加剧维护负担。

#### 3.3.3 Action 扩展问题

- 当前 `Action` 虽可通过反射识别，但缺少显示名、分类、摘要模板、参数顺序等元数据。
- 现有动作更偏历史迭代的功能累积，通用流程编排仍缺少一批基础动作。
- 编辑器对 `Action` 的认知停留在“类型 + 参数列表”层面，无法形成高质量的卡片摘要与分类配置体验。

#### 3.3.4 示例体系问题

- 现有 XML 更像测试样本，不是面向使用者的结构化示例。
- 缺少按“基础能力 / Action 分类 / 综合流程”分层的示例组织。
- 示例未与编辑器模板、新动作展示、回归测试形成统一闭环。

## 4. 功能需求

### 4.1 XML 主数据源与规范化读写

**描述**: 保持 XML 为 `StepSystem` 的主数据源。编辑器内部允许重构，但所有编辑结果仍通过 XML 落盘。导入器继续兼容旧格式，导出器统一输出规范化格式。

**验收标准**:

- [ ] 编辑器不引入新的主数据资产格式。
- [ ] 导入器兼容旧 XML 中的 action 属性参数和 `<param>` 两类写法。
- [ ] 导出器统一将参数写为 `<param key="..." value="..." />`。
- [ ] 导入旧 XML 后重新导出，不丢失关键语义信息。

**优先级**: P0

### 4.2 Graph 可视化增强

**描述**: 升级节点与连线的图内信息表达，使使用者在不切换到右侧 Inspector 的情况下，也能快速理解图结构、关键动作和分支条件。

**验收标准**:

- [ ] 节点卡片直接显示 `节点ID`、`节点名称`、`执行模式` 和 `Action` 摘要。
- [ ] 节点可显示 `Start`、`Running`、`Completed`、`Warning`、`Error` 等状态标记。
- [ ] 连线图上可直接显示条件与优先级。
- [ ] 图校验结果能映射到节点或连线的视觉提示。
- [ ] 运行态下能在图上明确区分当前节点、已完成节点和已执行路径。

**优先级**: P0

### 4.3 Graph 编辑效率增强

**描述**: 增加高频编辑能力，降低复杂图维护成本。

**验收标准**:

- [ ] 支持节点复制、粘贴、重复。
- [ ] 支持多选节点对齐、均分或批量移动。
- [ ] 支持自动布局，至少提供一种适用于 DAG 的横向布局。
- [ ] `Action` 列表支持排序调整。
- [ ] 新增节点和新增 `Action` 支持搜索或分类选择，而不是只暴露原始类型名。

**优先级**: P0

### 4.4 运行时联调增强

**描述**: 将 `StepManager` 的运行信息更直观地映射回 `Graph`，帮助定位当前流程位置、等待原因和上下文变化。

**验收标准**:

- [ ] 可在图中识别当前运行节点。
- [ ] 可在图中识别当前节点内部正在执行的 `Action`。
- [ ] 可显示已执行路径或最近执行轨迹。
- [ ] 上下文面板可查看当前节点、当前 `Action`、关键上下文字段和最近事件。
- [ ] 前进、后退、跳转补齐等动作会在图上产生可感知反馈。

**优先级**: P0

### 4.5 Action 元数据体系

**描述**: 在现有 `StepParamAttribute` 基础上补充 `Action` 级别的描述信息，使编辑器能自动组织分类、摘要、搜索与参数表单。

**验收标准**:

- [ ] 每个可编辑 `Action` 可声明显示名、分类、说明、关键词和摘要模板。
- [ ] 编辑器中的 `Action` 选择器按分类展示，而不是只显示完整类型名。
- [ ] 节点卡片可基于元数据渲染 `Action` 摘要。
- [ ] Inspector 可基于元数据控制参数显示顺序和常用项展示。

**优先级**: P0

### 4.6 新增通用 Action

**描述**: 扩展一批面向通用流程编排的 `Action`，优先覆盖流程控制、上下文操作和对象状态控制等高频需求。

**首批建议动作**:

- `StepWaitSecondsAction`
- `StepSetContextValueAction`
- `StepRemoveContextValueAction`
- `StepDispatchEventAction`
- `StepInstantiatePrefabAction`
- `StepDestroyTargetAction`
- `StepSetParentAction`
- `StepSetPositionAction`
- `StepSetRotationAction`
- `StepSetScaleAction`
- `StepPlayParticleAction`
- `StepSetActiveAction`

**验收标准**:

- [ ] 首批新增动作可在编辑器中被发现、被配置、被导出、被导入。
- [ ] 每个新增动作均遵循 `BaseStepAction` 生命周期约束。
- [ ] 涉及跳转补齐的动作具备非阻塞 `Apply` 行为。

**优先级**: P0

### 4.7 框架桥接 Action 预留

**描述**: 在不扩大当前实现范围的前提下，为后续 `LWFramework` 框架能力桥接提供分类与扩展入口。

**预留动作方向**:

- `StepOpenUIViewAction`
- `StepCloseUIViewAction`
- `StepSwitchProcedureAction`
- `StepPlay3DAudioAction`
- `StepPreloadAssetAction`

**验收标准**:

- [ ] 元数据分类中存在“界面与框架”分组。
- [ ] 代码结构允许后续低成本扩充桥接动作。

**优先级**: P1

### 4.8 示例体系分层

**描述**: 新建面向使用者和回归测试的分层示例体系。

**示例层级**:

- 基础能力示例
- Action 主题示例
- 综合流程示例

**验收标准**:

- [ ] 新增基础能力示例，覆盖流程、条件、并行、上下文等核心能力。
- [ ] 新增 Action 主题示例，按分类组织常用动作。
- [ ] 保留至少一份教学流程型综合示例和一份通用编排型综合示例。
- [ ] 示例可作为编辑器模板入口或演示入口被快速访问。

**优先级**: P0

## 5. 技术与交互设计

### 5.1 总体架构

本次升级采用“外部 XML 真源 + 内部编辑器分层重构”的方案。

外部约束：

- XML 仍为唯一主数据源。
- `StepXmlLoader` 仍负责运行时加载。
- 编辑器导入导出仍以 XML 为输入输出。

内部拆分建议：

1. `XML 归一化层`
   - 负责把旧 XML 和新 XML 读成统一编辑域模型。
   - 负责将统一模型导出成标准 XML。
2. `Graph 视图模型层`
   - 负责节点摘要、连线标签、状态映射和错误提示。
   - 不直接关心 XML 原始结构细节。
3. `Inspector 表单层`
   - 负责节点、连线、动作参数配置与校验。
4. `Action 元数据层`
   - 负责分类、显示名、摘要模板、关键词和参数展示策略。
5. `运行时联调层`
   - 负责 `StepManager` 运行信息到编辑器 `Graph` 的状态映射。

### 5.2 Graph 节点设计

节点从“单标题节点”升级为“信息卡片节点”。

节点卡片建议包含以下元素：

- 标题区：`节点ID`
- 副标题区：`节点名称`
- 标签区：`Start`、`Serial/Parallel`
- 摘要区：显示前 2 至 3 条 `Action` 摘要，超出部分显示 `+N`
- 状态区：运行中、已完成、警告、错误

颜色语义：

- 当前运行节点：黄色强调
- 已完成节点：绿色强调
- 警告节点：橙色提示
- 错误节点：红色提示
- 普通节点：保持低干扰中性色

### 5.3 连线设计

连线在图上直接表达：

- 优先级
- 条件 Key
- 比较类型
- 条件值

表现建议：

- 无条件连线：基础样式
- 有条件连线：带标签样式
- 高优先级连线：使用更高对比度或更粗视觉表现
- 非法条件：直接标记为警告或错误

### 5.4 运行时联调设计

联调层从“当前节点高亮”升级为“执行可视化”。

联调信息至少包括：

- 当前节点 ID
- 当前 `Action` 名称或序号
- 最近执行轨迹
- 最近事件列表
- 关键上下文字段快照

其中上下文面板建议支持两种视图：

- 全量 JSON 视图
- 关键字段观察视图

关键字段观察视图用于显示类似 `mode`、`score`、`isVip` 等常用流程判断字段，避免在大段 JSON 中难以定位。

### 5.5 Action 元数据设计

当前 `StepParamAttribute` 负责参数绑定，本次新增 `Action` 元数据负责编辑器展示语义。

建议引入动作描述结构，至少包含：

- `DisplayName`
- `Category`
- `Description`
- `Keywords`
- `SummaryTemplate`
- `DefaultExpanded`
- `CommonParameterKeys`
- `AdvancedParameterKeys`

分类建议如下：

- `流程控制`
- `上下文`
- `对象控制`
- `动画与特效`
- `音频`
- `界面与框架`
- `调试`

### 5.6 Action 参数展示设计

Inspector 对 `Action` 的参数展示遵循以下原则：

- 先展示常用参数，再展示高级参数。
- 参数顺序由元数据控制，不再完全依赖反射字段顺序。
- 对无效参数值直接给出编辑期提示。
- 允许保留未识别参数，避免旧 XML 导入后信息丢失。

### 5.7 示例体系设计

示例目录仍可放在 `Assets/0Res/RawFiles/`，但组织方式从“测试文件堆叠”升级为“能力分层”。

建议增加以下示例：

- `StepExample_BasicFlow.xml`
- `StepExample_ConditionBranch.xml`
- `StepExample_ParallelActions.xml`
- `StepExample_ContextOps.xml`
- `StepExample_ObjectControl.xml`
- `StepExample_Actions_Context.xml`
- `StepExample_Actions_Object.xml`
- `StepExample_Actions_AudioFx.xml`
- `StepExample_Flow_TeachingDemo.xml`
- `StepExample_Flow_GeneralPipeline.xml`

同时建议将 `StepProcedure` 演示入口升级为统一示例浏览入口，而不是写死单个 XML 路径。

## 6. 关键设计决策

| 决策 | 选择 | 备选方案 | 选择理由 |
|------|------|---------|---------|
| 主数据源 | XML | `Graph Asset` / 双轨持久化 | 用户明确要求 XML 继续作为主数据源 |
| 升级策略 | 方案 2：外部保留 XML，内部重构编辑器模型 | 方案 1：直接叠加功能；方案 3：平台化大升级 | 在可重构前提下，方案 2 在可落地性与长期可维护性之间最平衡 |
| 目标定位 | 通用流程编排 | 教学流程专用 / 剧情演示专用 | 需要兼容多类流程，避免系统过早业务化 |
| 兼容策略 | 导入兼容旧 XML，导出统一新规范 | 完全兼容不收口 / 完全断旧格式 | 既保留历史数据，又逐步收口规范 |
| Action 扩展顺序 | 先通用动作，后框架桥接动作 | 先做业务桥接动作 | 优先提高系统通用性和自洽性 |

## 7. 测试、迁移与风险

### 7.1 测试策略

本次升级至少覆盖以下四类验证：

1. XML 兼容与规范化
   - 旧 XML 可导入
   - 导入后的内部模型正确
   - 导出结果为统一新格式
   - 再导入后语义保持一致
2. 编辑器与 Graph 表现
   - 节点摘要、连线标签、状态映射正确
   - 复制粘贴、布局、排序等编辑操作不破坏数据
3. 运行时行为
   - 新增 `Action` 的参数绑定、生命周期和 `Apply` 语义符合预期
   - 条件分支、并行执行、跳转补齐不回归
4. 示例回归
   - 核心示例能正确导入导出
   - 重点综合示例可启动并完成关键流程

### 7.2 迁移策略

- 导入时兼容旧格式 XML。
- 导出时统一收口到规范化 `<param>` 形式。
- 允许编辑器内部重构，但不引入新的主持久化资产格式。
- 历史 `StepActionTest.xml`、`StepStage4Test.xml` 继续保留，作为兼容样例。
- 新示例作为主示例体系逐步替代历史“测试型示例”的展示职责。

### 7.3 风险与缓解

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| 编辑器内部重构导致现有窗口逻辑回归 | 中 | 高 | 先建立归一化模型与测试，再替换 UI 表现层 |
| Graph 信息量增加后节点视觉过载 | 中 | 中 | 节点摘要控制在前 2 至 3 条动作，详细信息仍放 Inspector |
| 新增 Action 数量较多导致首轮范围失控 | 中 | 高 | 首轮只实现通用动作主集，框架桥接动作作为第二优先级 |

## 9. 需求问答与方案确认记录

### 9.1 需求确认摘要

| 方案 | 核心思路 | 优势 | 劣势 | 结果 |

## 10. 验收标准

- XML 继续作为主数据源，编辑器和运行时围绕 XML 闭环工作。
- `Graph` 在图上即可直接表达节点摘要、连线条件、优先级和运行态信息。
- 编辑器高频操作明显优于当前版本，适合维护中型以上 DAG。
- 新增 `Action` 能通过元数据自动接入编辑器展示与配置流程。
- 示例从“历史测试文件”升级为“基础能力 + Action 主题 + 综合流程”的分层体系。
- 设计允许后续持续扩展，而不会再次把职责堆回 `StepEditorWindow` 单文件。
